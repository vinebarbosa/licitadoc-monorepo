# AGENTS.md

## Project Overview

- `licitadoc` is a `pnpm` + Turbo monorepo with:
  - `apps/web`: a Vite/React frontend
  - `apps/api`: a Fastify backend
  - shared packages for TypeScript config and an OpenAPI-based API client
- The backend follows a simple modular functional architecture:
  - Fastify for HTTP composition
  - modules by domain (`users`, `organizations`, `processes`, `documents`)
  - one file per use case
  - policies per module
  - plugins for infrastructure concerns
- The frontend is currently a minimal React Router app prepared for `shadcn/ui`, but `shadcn` has not been fully installed/generated yet.

## Project Structure

### Root

- `package.json`: root scripts for Turbo (`dev`, `build`, `lint`, `typecheck`, `format`)
- `pnpm-workspace.yaml`: workspace definition for `apps/*` and `packages/*`
- `turbo.json`: task graph and caching behavior
- `tsconfig.base.json`: shared TypeScript base
- `ARCHITECTURE.md`: backend architecture specification
- `README.md`: high-level project setup notes

### Apps

- `apps/web`
  - Vite app using React 19 and React Router
  - current source lives in `src/`
  - route files currently live in `src/routes`
  - global styles are in `src/styles.css`
  - `components.json` exists to support future `shadcn/ui` usage

- `apps/api`
  - Fastify app using:
    - `src/app` for bootstrapping
    - `src/plugins` for global infrastructure
    - `src/authorization` for global auth concepts (`Role`, `Actor`)
    - `src/modules` for domain modules
    - `src/db/DATABASE.md` for the database specification and business rules behind the schema
    - `src/db/schema` for Drizzle schema definitions
    - `src/shared` for reusable auth helpers, errors, HTTP helpers, and generic utils
    - `openapi/` for generated OpenAPI artifacts consumed by `packages/api-client`
    - runtime OpenAPI available at `/openapi.json`

### Packages

- `packages/api-client`
  - OpenAPI-driven client package
  - Kubb config lives in `kubb.config.ts`
  - source artifact lives in `apps/api/openapi/openapi.json`
  - custom fetch client lives in `src/client.ts`
  - generated output is expected under `src/gen`
  - `src/gen` is a generated artifact and should stay out of version control
  - this package is intended to generate:
    - typed client functions
    - TanStack Query hooks

- `packages/config-typescript`
  - reusable TS presets for `base`, `node`, and `vite`

## Development Workflow

### Install dependencies

```bash
pnpm install
```

### Run the monorepo

```bash
pnpm dev
```

- This delegates to `turbo dev`
- Expected app-level dev commands:
  - web: `vite`
  - api: `tsx watch src/app/server.ts`

### Build everything

```bash
pnpm build
```

### Typecheck everything

```bash
pnpm typecheck
```

### Lint

```bash
pnpm lint
```

Note:
- linting is handled by Biome
- generated folders such as `packages/api-client/src/gen`, `apps/api/postman`, and `apps/api/src/db/migrations` are ignored by the root Biome config

### Format

```bash
pnpm format
```

```bash
pnpm format:check
```

### Generate API client

```bash
pnpm --filter @licitadoc/api-client generate
```

Important:
- generated output is expected in `packages/api-client/src/gen`
- `packages/api-client/src/gen` is gitignored and must be regenerated instead of hand-maintained
- the OpenAPI artifact is generated by `apps/api`
- the API also exposes the current document at `/openapi.json`
- update route schemas first, then regenerate the client

## Code Conventions

### General

- Use TypeScript everywhere.
- Use ESM modules (`"type": "module"` is set in apps/packages).
- Prefer named exports.
- Follow existing formatting:
  - semicolons
  - double quotes
  - concise functions
- Use Biome as the formatter/linter source of truth for JS/TS/JSON files.

### Frontend conventions

- Put route-level code under `apps/web/src/routes`.
- Keep app composition simple:
  - `main.tsx` boots React
  - `router.tsx` defines route tree
  - route components live in `src/routes`
- Use workspace packages when logic/types are reusable:
  - `@licitadoc/api-client`

### Backend conventions

- Keep the Fastify structure intact:
  - `src/app`: boot and server startup
  - `src/plugins`: global infra
  - `src/modules`: domain logic
  - `src/shared`: reusable non-domain helpers
  - `src/db/schema`: Drizzle schema only
- Each module should follow the current pattern:
  - `routes.ts`
  - one file per use case, e.g. `get-user.ts`, `create-process.ts`
  - `<module>.schemas.ts`
  - `<module>.policies.ts`
- Route handlers should:
  - parse request data
  - get the authenticated actor
  - call use-case functions
  - avoid embedding domain logic directly
- Use-case files should:
  - receive explicit inputs
  - use `db` directly when needed
  - call module policies
  - return data or throw typed app errors
- Policies should stay inside the owning module.
- Global authorization concepts only belong in `src/authorization`.
- OpenAPI should be derived from route schemas, not maintained as a manual source file.

### Error handling

- Reuse `AppError` and specific subclasses from `apps/api/src/shared/errors`.
- Do not return ad hoc error shapes from modules when a typed error class is more appropriate.

### Auth conventions

- Authentication helpers belong in `apps/api/src/shared/auth`.
- Current session retrieval is centralized in `get-session-user.ts`.
- Better Auth is present but still scaffolded, not fully integrated.

## AI Agent Guidelines

### How to navigate the repo

- Start at the root:
  - check `package.json`
  - check `turbo.json`
  - inspect the relevant app/package
- For backend work, read:
  - `ARCHITECTURE.md`
  - `apps/api/src/app/build-app.ts`
  - `apps/api/src/db/DATABASE.md` before changing anything under `apps/api/src/db`
  - the target module folder
- For frontend work, read:
  - `apps/web/src/router.tsx`
  - `apps/web/src/routes/*`
  - shared packages if the change should be reused

### Where to create new code

- New frontend screens/routes:
  - `apps/web/src/routes`
- New frontend app utilities:
  - keep local to `apps/web/src` unless reuse across apps is clear
- New backend features:
  - create or extend a module under `apps/api/src/modules`
- New backend contract changes:
  - update route schemas in the owning module
  - regenerate `apps/api/openapi/openapi.json`
- New backend shared infra:
  - `apps/api/src/plugins` for global Fastify infrastructure
  - `apps/api/src/shared` for generic helpers
  - `apps/api/src/authorization` only for truly global auth types/concepts
- New database changes:
  - read `apps/api/src/db/DATABASE.md` first
  - keep table definitions in `apps/api/src/db/schema`
  - keep business rules mirrored between the specification and the Drizzle schema
- New shared UI:
  - keep it inside `apps/web` unless there is a strong cross-app need in the future
- New generated API client behavior:
  - configure in `packages/api-client`

### Where NOT to modify code casually

- Do not hand-edit generated Kubb output in `packages/api-client/src/gen` once it exists.
- Do not commit `packages/api-client/src/gen`; regenerate it locally/CI from the API contract.
- Do not put domain-specific backend logic into `apps/api/src/shared`.
- Do not bypass `build-app.ts` and register ad hoc Fastify boot logic elsewhere.
- Do not move backend auth rules into global folders if they are module-specific.
- Do not invent new top-level folders when an existing app/package already owns the concern.
- Do not change files under `apps/api/src/db/schema` without checking `apps/api/src/db/DATABASE.md` first.

### How to choose between extending a module vs creating a new one

- Extend an existing backend module when the feature belongs to the same domain entity/resource:
  - `users`
  - `organizations`
  - `processes`
  - `documents`
- Create a new module only when the domain is clearly distinct and deserves its own routes, schemas, policies, and use cases.
- On the frontend, extend existing route files only for small changes; create a new route component when the screen is distinct.

### Consistency rules

- Match the current backend naming style:
  - `get-user.ts`
  - `update-organization.ts`
  - `documents.policies.ts`
- Keep auth/permission decisions explicit via policies.
- Keep schemas in dedicated `*.schemas.ts` files.
- Prefer relative imports inside `apps/api/src` as the current code does.
- Prefer workspace packages for reused code rather than duplicating types/utilities.

## Testing Instructions

- There is currently no test runner or test suite configured in the repository.
- Do not claim tests were run unless you actually added and ran them.
- For now, the minimum verification path is:
  - `pnpm typecheck`
  - `pnpm lint`
  - package/app-specific build if relevant

Expectations for new code:
- New code should at least typecheck.
- If you add a testing framework later, follow the existing app/package boundaries rather than creating a global ad hoc test setup.
- If a feature is non-trivial and no tests are added, explicitly state that testing is still pending.

## Do / Don’t Rules

### Do

- Do preserve the monorepo split between `apps` and `packages`.
- Do add backend behavior inside the appropriate module.
- Do keep Fastify infrastructure in plugins.
- Do use typed app errors for backend failures.
- Do keep OpenAPI/client generation concerns inside `packages/api-client`.
- Do treat backend route schemas as the source of truth for the frontend/backend contract.

### Don’t

- Don’t add backend “service” or “repository” layers just because they are common elsewhere; this repo currently uses functional use-case files directly.
- Don’t place database queries inside `routes.ts` files.
- Don’t put module business rules in `shared/`.
- Don’t manually duplicate API response types in the frontend if they belong in generated client types.
- Don’t hand-maintain TanStack Query hooks that should be generated by Kubb once the OpenAPI flow exists.
- Don’t edit generated OpenAPI/client artifacts as the primary source; make the change in backend route schemas first.
- Don’t assume `shadcn/ui` components already exist just because `components.json` is present.
- Don’t assume Better Auth is fully wired; it is currently scaffolded.

### Good examples in this repo

- Good backend separation:
  - `apps/api/src/modules/users/routes.ts` calls use cases
  - `apps/api/src/modules/users/get-user.ts` holds operation logic
  - `apps/api/src/modules/users/users.policies.ts` holds authorization rules
- Good Fastify composition:
  - `apps/api/src/app/build-app.ts`
- Good shared error pattern:
  - `apps/api/src/shared/errors/*`
- Good generated-client boundary:
  - `packages/api-client/kubb.config.ts`
  - `packages/api-client/src/client.ts`

### Bad practices for this repo

- Bad: creating `user.service.ts` + `user.repository.ts` by default for simple CRUD when the module already uses functional use-case files.
- Bad: adding a new domain directly under `apps/api/src/shared`.
- Bad: hardcoding frontend fetch logic in route components once `@licitadoc/api-client` is in use.
- Bad: editing generated client hooks manually instead of changing the OpenAPI spec or Kubb config.
