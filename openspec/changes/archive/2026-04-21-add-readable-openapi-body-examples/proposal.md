## Why

Some request body examples rendered by Scalar are still hard to read because the exported OpenAPI falls back to regex-heavy schemas, empty strings, or generic placeholder values for fields like UUIDs, slugs, names, and dates. That makes the docs less helpful exactly where API consumers most need copyable, realistic payload examples.

## What Changes

- Add explicit, readable OpenAPI examples for request body fields in application-owned route schemas where autogenerated examples are currently noisy or ambiguous.
- Introduce a shared pattern for common example types such as UUIDs, slugs, names, dates, and other text inputs used in request bodies.
- Backfill the affected request body schemas in modules such as invites, departments, organizations, and processes so Scalar shows realistic payload examples.
- Keep runtime validation, transforms, and endpoint behavior unchanged while improving only the generated contract examples.
- Preserve compatibility with the exported `/openapi.json` document and the existing client-generation workflow.

### New Capabilities

### Modified Capabilities
- `api-route-schemas`: Application-owned Zod route contracts must support explicit, readable examples for request body fields so the exported OpenAPI document does not fall back to regex-heavy or empty autogenerated payload examples.

## Impact

- Affected code: shared schema utilities and request body schema files in `apps/api/src/modules/**`.
- APIs: request body examples in Scalar/OpenAPI become more legible, but endpoint behavior is not intended to change.
- Systems: backend maintainers and API consumers get more realistic payload examples in the generated docs and derived tooling.
