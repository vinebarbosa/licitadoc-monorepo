## ADDED Requirements

### Requirement: Application-owned request body schemas MUST expose readable examples
The API MUST allow application-owned Zod-backed request body schemas to declare explicit, readable examples for common input fields so the exported OpenAPI document does not fall back to regex-heavy or empty autogenerated payload values.

#### Scenario: Documenting a request body with UUID and text inputs
- **WHEN** a route body in `apps/api` includes fields such as UUIDs, slugs, names, dates, or descriptive text
- **THEN** the exported OpenAPI document includes human-readable examples for those fields
- **AND** Scalar renders a realistic request payload instead of raw regex-driven or empty placeholder values

#### Scenario: Documenting a transformed request body field
- **WHEN** a request body field is built from a refined or transformed Zod schema
- **THEN** the readable example is attached to the final schema exported in the route contract

### Requirement: Readable request body examples MUST remain compatible with the exported contract
The API MUST preserve explicit request body examples in `/openapi.json` without breaking the existing contract export and typed client generation workflows.

#### Scenario: Regenerating contracts after adding body examples
- **WHEN** the API exports or serves `/openapi.json` after request body example metadata is added
- **THEN** the document contains the explicit readable examples for the affected request body fields
- **AND** the existing client-generation workflow can still consume the exported document
