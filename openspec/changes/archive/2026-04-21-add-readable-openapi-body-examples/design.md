## Context

`apps/api` derives its OpenAPI document from Zod-backed route contracts, and Scalar renders request payload examples directly from that exported schema. After the email-example improvement, several request bodies still fall back to low-signal autogenerated output such as raw UUID regexes, empty strings, or generic placeholders for fields like slugs, names, dates, and organization references.

The current codebase already has a small shared helper for explicit OpenAPI examples, which shows that the right place to fix this is still the schema layer rather than the docs UI. The remaining issue is broader than email and spans multiple application-owned request body schemas in invites, departments, organizations, processes, and other editable resources.

## Goals / Non-Goals

**Goals:**
- Ensure request body examples shown in Scalar/OpenAPI are readable and realistic for common application-owned inputs.
- Reuse a shared convention for example metadata so modules do not scatter ad hoc literals.
- Apply examples to the final Zod schemas exported in request bodies, including transformed or refined string fields.
- Keep the exported `/openapi.json` compatible with the existing contract and client-generation workflows.

**Non-Goals:**
- Changing runtime validation, transforms, defaults, or endpoint behavior.
- Reworking response examples beyond what is strictly needed to support request body readability.
- Solving every possible autogenerated example issue for every schema in one pass.
- Customizing Scalar rendering behavior in the frontend instead of improving the source contract.

## Decisions

### Decision: Fix unreadable body examples at the Zod/OpenAPI schema layer
The unreadable payloads come from missing or weak example metadata in the exported contract, so the implementation should enrich the route schemas rather than override Scalar behavior.

Alternatives considered:
- Override or theme Scalar to hide schema details.
  Rejected because the underlying OpenAPI would still remain low quality for other tooling.

### Decision: Extend the shared example pattern beyond email
The API already uses a shared helper for explicit email examples. This change should expand that pattern to additional common request-body field types such as UUIDs, slugs, names, dates, and descriptive text.

Alternatives considered:
- Hardcode examples independently in each module.
  Rejected because it would duplicate conventions and make examples drift across modules.

### Decision: Attach examples to the final request-body schemas consumed by route contracts
Many body fields are built from trimmed, refined, transformed, or defaulted Zod schemas. The example metadata must live on the final schema shape that `fastify-zod-openapi` exports so Scalar receives the intended example instead of intermediary schema details.

Alternatives considered:
- Attach examples only to low-level helper fragments.
  Rejected because transforms and refinements can leave the final exported contract without the desired example metadata.

### Decision: Prioritize application-owned write endpoints with the most visible noisy examples
The immediate docs pain is on payload-heavy create/update routes such as invites, departments, organizations, processes, and similar bodies. Focusing there keeps the change useful and scoped.

Alternatives considered:
- Expand immediately to all query params, responses, and hidden auth routes.
  Rejected because the request is specifically about body readability and the smallest useful fix is in application-owned request payloads.

## Risks / Trade-offs

- [Example literals become inconsistent across modules] -> Centralize reusable examples and helper wrappers in the shared HTTP/schema layer.
- [Examples get attached before a transform/refinement and fail to appear in the exported contract] -> Apply metadata to the final request-body field schemas that route contracts actually expose.
- [Scope grows into a general OpenAPI content-polishing effort] -> Keep this change focused on noisy request body examples in application-owned routes.

## Migration Plan

Add shared request-body example helpers/constants, backfill the affected body schemas to use them, regenerate or inspect the exported OpenAPI document, and verify Scalar now shows readable payloads. No runtime migration is required because this is a documentation-contract improvement only.
