## Context

`apps/api` derives Swagger/OpenAPI from Zod-backed route contracts, and Swagger UI currently falls back to autogenerated examples when the schema does not provide explicit example metadata. For email-shaped fields, that fallback is producing extremely large values in the rendered docs, which makes request and response examples noisy and hard to read.

The API already uses Zod as the single source of truth for route contracts, and there is prior evidence that OpenAPI metadata can be attached directly to schemas through Zod metadata. This makes the problem a good fit for a contract-layer improvement rather than a Swagger UI customization.

## Goals / Non-Goals

**Goals:**
- Ensure email fields exposed by application-owned route schemas render concise, human-readable examples in Swagger/OpenAPI.
- Add a reusable pattern for applying email examples so modules do not each invent different literals.
- Keep the generated OpenAPI document compatible with the current export and client-generation workflows.

**Non-Goals:**
- Changing runtime validation or serialization behavior for email fields.
- Reworking the visual layout of Swagger UI itself.
- Solving every autogenerated example issue for every possible field type in the same change.
- Modifying hidden Better Auth proxy routes or third-party auth documentation beyond the application-owned schema layer.

## Decisions

### Decision: Fix the issue at the schema layer instead of at the Swagger UI layer
The oversized examples come from missing example metadata in the OpenAPI schema, so the fix should be to enrich the Zod-backed route contracts rather than override Swagger UI rendering.

Alternatives considered:
- Override Swagger UI example rendering with custom frontend logic.
  Rejected because the source problem is the generated contract, not the UI shell.

### Decision: Introduce a shared helper or shared schema pattern for email examples
Email examples should be deterministic and reusable. A shared helper in the existing HTTP/schema layer can centralize the example literals and avoid repeating `example` metadata inconsistently across modules.

Alternatives considered:
- Add raw example metadata independently in every schema file.
  Rejected because it would scatter the convention and drift quickly.

### Decision: Apply example metadata to the final email schemas used in OpenAPI contracts
Some email fields are built through transforms or refinements, so the implementation should ensure the example metadata is attached to the final schema shape that Swagger/OpenAPI sees.

Alternatives considered:
- Attach metadata only to intermediate helper schemas.
  Rejected because refinement/transform chains may not preserve the intended metadata location cleanly.

### Decision: Backfill the application-owned route schemas that expose email fields first
The immediate user-facing problem is visible in the application-owned Swagger routes, such as users, invites, and organization institutional email fields. That is the smallest useful scope that fixes the problem without expanding into unrelated docs surfaces.

Alternatives considered:
- Expand the change to every generated auth document immediately.
  Rejected because the request is already satisfied by fixing the app-owned schema layer, and the auth docs come from a different source.

## Risks / Trade-offs

- [Different modules may still choose inconsistent example strings] -> Centralize the convention in a shared helper or constant.
- [Metadata may be attached to the wrong stage of a transformed schema] -> Apply examples to the final schema shapes used in the exported route contracts.
- [The change could become a broader OpenAPI examples refactor] -> Keep this change scoped to email examples and readable Swagger output.

## Migration Plan

Add the shared email example pattern, update the affected route schemas to use it, regenerate or inspect the OpenAPI output, and confirm Swagger now shows readable email examples. No runtime migration or rollout sequencing is required because the change is contract-documentation only.
