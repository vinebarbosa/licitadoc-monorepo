## Why

The OpenAPI document currently allows Swagger UI to invent very large example strings for email fields when no explicit examples are provided in the Zod-backed route schemas. That makes the docs noisy, harder to scan, and less representative of how API consumers should actually use the contract.

## What Changes

- Add explicit, short, human-readable examples for email-shaped fields exposed by application-owned route schemas in `apps/api`.
- Introduce a reusable pattern for attaching OpenAPI example metadata to email schemas so route modules do not repeat ad hoc example literals inconsistently.
- Ensure the generated OpenAPI document and Swagger UI show concise email examples in request and response payloads where email fields are present.
- Keep the generated contract compatible with the existing client-generation workflow.

## Capabilities

### New Capabilities

### Modified Capabilities
- `api-route-schemas`: Route contracts derived from Zod schemas must support explicit, readable examples for email fields so the exported OpenAPI document does not fall back to oversized autogenerated values.

## Impact

- Affected code: shared schema utilities and route schema files that expose email fields in `apps/api/src/modules/**`.
- APIs: Swagger/OpenAPI examples become more readable, but no runtime endpoint behavior is intended to change.
- Systems: backend maintainers and API consumers get cleaner examples in the generated docs and derived tooling.
